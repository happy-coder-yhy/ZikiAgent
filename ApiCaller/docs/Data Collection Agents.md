## 边界条件
* 平台已经完成必要选项配置（物品库、标签库等），用户的采集任务涉及范围必定在该配置范围内
* 采集任务类型：
    * 真机采集
        * 采集设备采集到的数据会直接上传到云端系统
    * 视频采集
        * 所有基于EGO或互联网视频的数据采集方式均为视频采集，其与真机采集的差异在于采集到的数据只能保存到本地的某个工作站，需要后续带回后由管理员进行统一上传
* 管理员职责：
    * 创建采集任务，确认后发布
        * 任务类型为：
            * 具体任务：需要严格的任务模板，明确定义了任务的初始状态、步骤 （精调数据）
            * 半无感需求：定义了大致的任务类型、空间等标签 （居家等场景）
            * 无感需求：只定义场景，采集从开始到结束的超长视频（流水线工位）
    * 将回收的采集视频数据进行上传
        * 回收的视频数据已经按照文件夹与TaskID进行区分归类

* 采集员只负责领取采集任务，进行online采集或视频采集。
    * Online采集的过程无需Agent系统监控，采集员每次采集一条视频后都需要单独进行确认上传/删除操作，由采集设备自动记录并上传
    * 视频采集任务中的数据会自动归档到选择的对应作业下，但是在开始采集前，也需要采集员选中对应的采集任务

**待确认事项**
* 实际用户使用场景的边界划分
    * 平台物品库、标签库配置是否能够默认用户可以提供？
        * 手动导入：提供具体的参数配置文件的导入接口或规范
        * 默认生成：直接由Zata提供已经提前配置好的常用平台配置
        * 推理生成：【在信息量不足的前提下，这操作是否有其合理性？】
    * 管理员在使用时，是否一定是有具体的任务需求？还是也有可能只是一个非常简单的粗略需求？
        * 如果任务模板需要从粗略需求中自动生成的话，关于任务的标准步骤流程应该如何进行有效补全？由谁来进行评判？是否有可能涉及对LLM的后训练？
    * 采集员领取任务与作业的过程，能够给到什么程度的提示？
        * 我现在可以用XX设备进行采集，给我推荐一个合适的采集作业？
        * 给我推荐一个目前优先级最高的采集作业

## 阶段性建设目标
### 系统功能
* 基于Hermes搭建一套基础的Agent系统

* Headless API Calling：允许模型直接调用ZataAPICaller发送操作Zata Platform对应的API请求，并判断请求调用是否成功，以及失败原因反馈。

* Auto-Zata状态管理：
    * 在每次用户登录、执行API后，调用userinfo等接口同步Zata平台关键配置信息
        * 获取当前用户配置下的物品库、标签库等配置信息
        * 获取所有待采集任务、作业的数量、进度
    * 能够确定自身处于操作流程的哪个环节，具备从某个状态退回到初始状态的能力

当前本项目已实现的第一步是 `ZataAPICaller.sync_platform_configuration(pageSize=200)`：可同步当前登录用户可访问的项目、任务、标签分类、标签树、扁平标签索引、物品目录树和具体物品。本项目只提供 API Caller Toolkit，不负责自动推断或补全任务字段。

* 采集任务配置表单生成
    * 管理员通过用户交互自动生成Online采集任务/视频采集任务 表单，由用户确认后下发； 
        * 项目列表/物品库/标签库为空，或缺少相关配置时，主动尝试根据采集任务需求进行补充完善，但完善方式需用户确认
        * 【作业组合剪枝】

* 采集任务检索：
    * 采集员通过用户交互快速选择到合适的Online采集作业，并了解当前采集进度，确认后开始采集；
    * 采集员通过用户交互快速选择到合适的视频采集作业，并确认后上传视频数据；

* Failure Handling:
    * 采集任务表单，具备合规性检测能力，不合规冲突在多轮尝试修复失败后，交由用户手动配置冲突位置；

### 用户交互
接入大模型来实现一个基础的Agent，其可以根据用户的输入进行意图理解，并调用对应的API。

目前来看，其实管理员的操作暂时不会涉及到连续操作多个不同的API，而是每次输入都只会明确指向某一个API请求，其实际调用过程大概会有以下几种情况：
- 用户输入： 帮我新建一个名为 ABC 的项目。  此时Agent会自动调用create_project的请求，并返回结果（是否成功或失败）
- 用户输入： 帮我创建一个数据采集任务
    - 提供详细的数据采集需求文档：可以直接确定是严格任务还是半无感任务，具体的任务配置清单
    - 只简要口头描述，此时可能需要额外的问答交互来确定创建模式，或直接明确 创建严格任务 和 创建任务 是指代两种完全不同的创建方式

- 系统配置查询
  - 现在都有哪些项目
  - 现在都有哪些任务
  - xx任务下都有哪些作业？
  - 现在采集进度最快/最慢 的作业是哪个

  - 现在的 场景 / 空间 标签都有哪些

当前 API Caller Toolkit 已覆盖一批任务管理员相关平台操作。后续上层 Agent 或 workflow 项目可以基于这些结构化调用自由组合流程。POST/PUT/批量删除的公开调用边界应来自 OpenAPI 文档，不应要求上层调用者手写完整 payload。




## 实际应用场景流程描述
对于整个数字采集平台，其用户只分为 任务管理员 和 采集员 两类，而采集任务则分为下发后需要通过EGO设备现场开始采集的online采集任务，以及视频采集任务两类。

对于Online采集任务，其具体操作流程如下：
1. 任务管理员在登录平台后（尤其是首次登录）需要先对平台的项目列表、物品库、场景标签、空间标签、动作标签、设备标签进行配置；
2. 根据任务需求创建一条采集任务，采集任务可以通过任务模板，或直接创建。
在创建任务（或任务模板）时需要指定其对应的项目名、采集设备、场景、空间、动作以及涉及物品，并且定义

任务初始状态

以及采集数据的动作步骤：

```
如对于水果物品扫码任务，需要5个步骤：
1）将{水果}从{扫码台}左侧{移动}到{扫码台}中间；
2）{抓取}{扫码枪}；
3）将{扫码枪}{对准}{水果}；
4）将{水果}移动到{扫码台}右侧；
5）{放下}{扫码枪}；
```

3. 通过涉及的被扫码物品的排列组合，一条任务下会有多个不同的作业，每个作业有明确的采集数据条数；
4. 管理员确认任务内容无误后，选择“发布”任务；
5. 采集员在登录平台后，领取某一个online采集任务，选择任务下的某一条采集作业选择”开始采集“；
6. 采集员到采集设备上登录自身帐号，并点击“开始采集”，一条数据采集完成后，选择“完成采集”，此时该数据会自动上传至数据平台的对应作业下；
7. 直至某作业数据条数达到要求，此时设备会进行提示。此时需要采集员在数据平台上重新选择一条作业继续采集。

对于视频采集任务，其操作流程如下：
1. 管理员在登录平台后（尤其是首次登录）需要先对平台的项目列表、物品库、场景标签、空间标签、动作标签、设备标签进行配置； 
2. 管理员创建一个上传采集任务，采集任务分为以下三类：
* 严格任务（定义方式与Online采集任务一致）
* 场景任务，只限定任务场景，每条采集视频中可能包含多种不同的任务过程
* 指令任务，只限定场景，每条采集视频有明确的开始、结束信号。
3. 创建一系列具体作业，管理员确认任务无误后下发；
4. 采集员在登录平台后，领取某一个视频采集任务，并开始采集作业，其采集方式也分为两类：
4.1 选择与Online采集类似的方式，佩戴EGO设备现场采集并上传数据
4.2 选择将所有采集视频汇总后，再统一上传  （采集视频可能来自互联网等其他渠道）


## 总体技术方案
### Agent需求拆分
对于Multi-Agent系统来说，为了将上述流程进行用户无感化，其实是需要分别相应任务管理员和采集员的操作需求。

* 任务管理员
    * 可以根据需求文档自动初始化数据平台，完成项目、物品库、标签库的创建更新；
    * 可以根据简要描述快速自动生成对应的待采集任务模板，并创建采集任务进行下发；
    * 可以在使用过程中提示目前可能缺失的物品库、标签库信息，甚至是存在冲突、重复的配置信息；
    * 可以提示任务管理员不同采集任务的重要程度、优先级（基于现有数据库存比例、平均数据采集速度进行分析推荐）

第一阶段已完成任务管理员 API 操作基础层：登录、用户信息、平台配置读取/创建/更新/删除、任务创建/更新/发布/归档/删除、job 创建/更新/删除、真实平台只读配置联调。当前正在将项目范围收敛为核心 `ZataAPICaller`，尚未实现自然语言任务模板生成、自动补全、缺失配置自动创建和优先级分析。
* 采集员
    * 可以根据简要描述，快速匹配待采集的作业；
    * 可以直接推荐下一步可以采集的作业
    * 可以分析放置数据的文件夹，自动匹配对应的采集作业进行上传

### 核心工具
为了使得上述无感操作的实现，首先需要将涉及的各类操作工具进行实现，例如：
* 可以以无头模式直接调用Zata Platform的各个操作API接口
    * 自动登录平台
    * 读取当前项目列表、物品库、标签库等信息；以及任务列表、作业进度等信息；
    * 创建、更新、{删除} 物品库、标签库等平台配置信息；
    * 创建Online采集任务模板
    * 创建视频采集任务

已实现的无头 API 能力集中在 `modules.api_caller.ZataAPICaller`。其中 `sync_platform_configuration` 是后续 Agent 在创建任务前准备 ID/code 上下文的入口；其读取物品库时会先展开 `object-categories` 树，再按每个 `categoryId` 查询 `object-items`。

* 可以读取本地文件
    * 数据采集需求文档
    * 用户语音转文本，简要任务描述
    * 已采集视频
